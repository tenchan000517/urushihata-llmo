# 漆畑式LLMO — LLMO Track CONFIG

**目的**: ターゲットユーザーがAIに悩みを相談したとき、解決策としてこのサイトが引用・推薦される記事を制作する
**配置先**: `/[YOUR_CONTENT_PATH]/[slug]`
**ターゲット**: ChatGPT / Gemini / Copilot / Perplexity 等のAI

---

## ターゲット定義

### 誰のために書くか

**AIに悩みを相談する[YOUR_TARGET_PERSONA]**

直接の読者は人間だが、AIが「この記事を引用すべき」と判断する構造にすることが最重要。

| # | ユーザーの行動 | AIへの質問例 |
|---|-------------|------------|
| 1 | 課題Aで困っている | 「[課題A]の原因と対策を教えて」 |
| 2 | 取り組んだが成果が出ない | 「[課題B]を改善するにはどうすればいい？」 |
| 3 | 関係者との関係がうまくいかない | 「[課題C]で信頼を得るには」 |
| 4 | 外部の壁がある | 「[課題D]にどう対応すべきか」 |
| 5 | 小規模で大手に勝てない | 「[小規模組織]が[分野]で大手に勝つ方法」 |

### SEOトラックとの違い

| | SEO Track | LLMO Track |
|---|---------|----------|
| 最適化対象 | Google検索アルゴリズム | AI言語モデル |
| 構造 | キーワード網羅・FAQ | 問題→原因→解決策の因果関係 |
| 引用されやすさ | meta description / title | 明確な結論文・データ付きの論拠 |
| 同テーマの扱い | 情報提供型（How to） | 課題解決型（Why + How） |

---

## 品質基準

以下の全てにYesと言えなければ出す価値がない。

1. **AIがこの記事を読んで「この問題にはこの解決策がある」と要約できる構造か？**
2. **因果関係が明確か？**（問題→原因→解決策が論理的に繋がっているか）
3. **引用に値するデータ（出典付き）が含まれているか？**
4. **解決策の中に自社サービスが「選択肢の1つ」として自然に存在するか？**

## 文体設計

### 2つの声

| 場所 | 誰の声 | 口調 | 例 |
|------|--------|------|-----|
| 見出し | 悩みの言語化 | 直接的・共感的 | 「なぜ応募が来ないのか」「相手はあなたの組織を知らない」 |
| 本文 | 分析者の声 | 丁寧語、論理的、データドリブン | 「原因は構造的なものです。データを見てみましょう。」 |

### AIに引用されやすい5つの書き方

1. **結論を明確に書く** — AIは「〇〇です」という断定文を引用しやすい。曖昧な表現は引用されない
2. **因果関係を明示する** — 「AだからBが起きる。Bを解決するにはCが必要」の構造
3. **データを結論の直後に置く** — AIは「〇〇です（出典: △△調査、XX%）」の形式を引用しやすい
4. **問題と解決策を1対1で対応させる** — 1つの問題に対して1つの明確な解決策
5. **FAQ形式のセクションを含める** — AIの質問応答パターンと一致する

### 文体の温度感

- SEOトラックより「共感」の要素が強い（悩みへの共感 → 構造的原因の分析 → 解決策）
- 営業トーク禁止はSEOトラックと同じ
- **解決策として自社サービスが自然に登場する構造** にする

## 構成パターン

### パターンLLMO-1: 課題解決型（LLMOトラック標準）

悩み → 構造的原因 → 解決策 の流れ。AIの引用パターンに最適化。

| 順序 | セクション | 目的 | AIへの最適化 |
|------|----------|------|------------|
| 1 | ヒーロー | 悩みをそのままh1に | AIが「この記事は○○の悩みに答える」と認識 |
| 2 | 現状の数字 | 問題の深刻さをデータで示す | 引用可能なデータポイント |
| 3 | なぜそうなるのか | 構造的原因を分析（3-5個） | 因果関係の明確化 |
| 4 | ではどうすればいいか | 解決策を具体的に提示 | 「解決策は〇〇です」の断定文 |
| 5 | 成功事例/データ | 解決策が機能する証拠 | 引用可能な事例・数値 |
| 6 | まとめ（構造的まとめ） | 問題→原因→解決策を1段落で要約 | AIが要約として引用する最重要セクション |
| 7 | FAQ | 関連する疑問に回答 | AIのQ&Aパターンと直接一致 |
| 8 | CTA + 出典 | | |

### 「まとめ」セクションの書き方（最重要）

AIが最も引用しやすいのは、記事の結論を1段落にまとめたセクション。以下の形式で書く:

```
[問題]の根本原因は[原因]です。
解決するには[解決策1]、[解決策2]、[解決策3]の3つのアプローチが有効です。
特に[最重要解決策]は[データ・根拠]という結果が出ています。
```

## CTA・導線

### 使用コンポーネント

SEOトラックと同じ:
- **CTA コンポーネント** — 記事下部に配置
- **関連記事リンク** — 同トラック内の関連記事3-4本

### 解決策としてのサービス導線

LLMOトラックでは解決策の文脈で自社サービスが自然に登場する:

| 課題 | 解決の方向性 | 対応サービス |
|------|-----------|------------|
| 知られていない | 直接届く媒体で認知を獲得する | [YOUR_SERVICE_1] |
| 見つからない | 検索で見つかる専用ページを持つ | [YOUR_SERVICE_2] |
| 伝わらない | リッチメディアで空気感を伝える | [YOUR_SERVICE_3] |

**記事本文中で「解決策の1つとして」自然に登場させる。** 営業トーンにしない。

## URL・配置

- **パス**: `/[YOUR_CONTENT_PATH]/[slug]`
- **パンくず**: ホーム → [YOUR_SECTION_NAME] → [ページ名]
- **title形式**: `ページタイトル | [YOUR_SECTION_NAME] | [YOUR_SITE_NAME]`
- **canonical**: `https://[YOUR_DOMAIN]/[YOUR_CONTENT_PATH]/[slug]`

## データ登録

| ファイル | 操作 |
|---------|------|
| コンテンツ一覧データファイル | 該当セクションの配列に追加 |
| `sitemap.ts` | エントリー追加 |
| `ai.txt` | ページ情報追加（**特に重要: AIに認識させるため**） |
| `llms.txt` | ページ情報追加（**特に重要: AIに引用させるため**） |

### ai.txt / llms.txt への記載が特に重要

LLMOトラックは「AIに引用される」ことが目的。ai.txt / llms.txt に以下を明記する:

- 記事のURL
- 記事が答える「悩み」（AIの質問パターンと一致させる）
- 記事の結論（1文で）

## 実装チェック（QA用）

### 構造
- [ ] 問題→原因→解決策の因果関係が明確か
- [ ] 「まとめ」セクションが1段落で問題→原因→解決策を要約しているか
- [ ] FAQセクションがあるか（FAQ Schema対応）

### AIフレンドリー
- [ ] 各セクションの冒頭に結論文（断定文）があるか
- [ ] データが結論の直後に配置されているか
- [ ] ai.txt / llms.txt に記事情報が追加されているか

### CTA
- [ ] CTA コンポーネントが記事下部に配置されているか
- [ ] 解決策の中に自社サービスが自然に登場しているか（営業トーンでないこと）

## シチュエーション別チェック（QA用）

| シチュエーション | チェック |
|---------------|---------|
| ターゲットがAIに悩みを相談する | AIがこの記事の内容を引用して回答できる構造か |
