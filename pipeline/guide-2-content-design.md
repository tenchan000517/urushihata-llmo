# 漆畑式LLMO Pipeline — Step 2: コンテンツ設計 / Content Design

**目的: リサーチ結果を受け取り、誰に・何を・どの順で伝えるかを設計する。**

**トラック固有の設定はトラック設定ファイル（CONFIG）を参照すること。**

---

## このガイドの使い方

1. リサーチ済みデータを受け取ったら、**まずトラック設定ファイル（CONFIG）を読む**
2. ターゲット定義・品質基準・構成パターンを CONFIG から取得する
3. このガイドの共通ルールに従って構成案を設計する
4. 構成案を出力フォーマットに従ってまとめ、ユーザーに提示する
5. ユーザー承認後に次工程（執筆）へ渡す

**判断に迷ったらユーザーに聞く。「おそらくこうだろう」で進めない。**

---

## 共通の設計原則

### 全トラック共通の品質基準

- **読んだターゲットが「次に何をすればいいか」わかること。** これが唯一の品質基準
- **抽象的な「考え方」ではなく具体的な手順。** 読んだら動ける。迷わない
- **「記事」ではなく「コンテンツ」。** テンプレートを機械的に埋めたものは出す価値がない
- **出典のない情報は存在しないものと同じ**

### トラック設定ファイル（CONFIG）から取得するもの

| 項目 | CONFIG のセクション |
|------|---------------------|
| ターゲット定義（誰のために作るか） | `## ターゲット定義` |
| 品質基準の問い | `## 品質基準` |
| 構成パターン | `## 構成パターン` |
| CTA/導線の設計 | `## CTA・導線` |
| ページのURL構造 | `## URL・配置` |

---

## 構成設計の手順

### Step 1: ターゲットを確認する

CONFIG の「ターゲット定義」を読み、**このページはどのターゲットに一番刺さるか** を決める。全員に万遍なく当たるコンテンツは、誰にも刺さらない。

### Step 2: 「読んだ後に『できる』こと」を1文で書く

これが構成案の最重要項目。曖昧なら構成全体がぶれる。

### Step 3: 構成パターンを選択する

CONFIG に定義されたパターンから最適なものを選ぶ。パターンに当てはまらない場合はユーザーに相談する。

### Step 4: セクション構成を設計する

各セクションに「目的」と「内容概要」を明記する。目的のないセクションは削除する。

### Step 5: 品質基準の問いに全てYesと言えるか確認する

CONFIG の「品質基準」のチェック項目を全て確認する。

---

## 構成案の出力フォーマット

```
## 構成案: [ページタイトル]

**トラック**: SEO / LLMO / User
**URL**: /[path]/[slug]
**使用パターン**: [パターン名]
**テーマ色**: [色名]（該当する場合）

### ターゲット
[誰に向けたページか。CONFIG のターゲット定義から選択]

### 読んだ後に「できる」こと
[1文で。このページを読んだターゲットが具体的に何ができるようになるか]

### セクション構成

| # | セクション名 | 目的 | 内容概要 |
|---|------------|------|---------|
| 1 | ヒーロー | ... | タイトル案 + サブテキスト案 |
| 2 | ... | ... | ... |

### 出典の使い方
[リサーチ結果のどのデータをどのセクションで使うか]

### 特記事項
[このページ固有の注意点]
```

---

## 構成設計のチェックリスト（共通）

- [ ] ターゲットの現実から離れていないか
- [ ] 一般論になっていないか
- [ ] 手順と感情の両方があるか（該当する場合）
- [ ] 「次に何をすればいいか」が明確か
- [ ] CONFIG の品質基準の問いに全てYesと言えるか
- [ ] 1つのカードに情報を詰め込みすぎていないか
- [ ] 出典は全て明記される設計になっているか
