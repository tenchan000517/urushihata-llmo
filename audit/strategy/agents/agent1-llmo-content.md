# Agent 1: LLMO/AI引用コンテンツ分析

## 目的

2つの問いに答える:
1. ユーザーがAIに悩みを相談したとき、**あなたの記事が引用されるか**（情報源としての認知）
2. 「この悩みを相談できる企業は？」と聞かれたとき、**あなたの組織が推薦されるか**（サービス提供者としての認知）

**1だけでは不十分。2ができなければAIの回答にあなたの組織は出ない。**

---

## 調査手順

### Step 1: LLMOトラック記事のインベントリ

LLMO/AI引用を狙った記事を全て特定し、以下を記録する:

| ID | タイトル | URL | 対象の悩み | 言及されるサービス | まとめセクションの有無 | FAQの有無 |
|----|---------|-----|----------|-----------------|-------------------|----------|

**全記事を読むこと。サンプリングNG。**

### Step 2: 悩みカバレッジ分析

#### 2-1. カバー済みの悩み一覧
記事がカバーしているユーザーの悩み（AIへの質問パターン）をリスト化。

#### 2-2. 未カバーの悩み候補

**ソース1: Search Consoleデータ**
悩み系クエリを抽出（「○○ 対策」「○○ 解決」「○○ 難しい」等）

**ソース2: トラック設定ファイルのターゲット定義**

**ソース3: AIが実際に回答する悩みのパターン**
ユーザーがAIに聞く典型的な質問を列挙し、それぞれに対応する記事があるか確認

### Step 3: LLMO構造品質チェック

各記事について以下を確認:

| チェック項目 | 基準 |
|------------|------|
| 断定文で結論を書いているか | AIは断定文を引用しやすい |
| 結論の直後にデータがあるか | 根拠付き断定がAI引用の条件 |
| 「まとめ」セクションがあるか | AIが最も引用しやすい箇所 |
| まとめに問題→原因→解決策があるか | 1段落で因果関係が完結 |
| FAQ（構造化データ付き）があるか | リッチリザルト + AI回答ソース |
| ai.txt に記載があるか | AIクローラーへの明示的案内 |
| llms.txt に記載があるか | LLMへの推薦根拠 |

### Step 4: サービス推薦力の評価（最重要）

**AIが「悩み解決ができる企業」として推薦するために必要な要素:**

1. **エンティティ認識**: 構造化データでOrganizationとして認識されているか
2. **サービスと悩みの接続**: 記事内で悩み→解決策→サービスが自然に繋がっているか
3. **独自性の主張**: 他社にない構造的優位性が明記されているか
4. **地域との紐付け**: 「[地域]で[サービス]」のパターンで発見できるか

### Step 5: 競合比較

同じ悩みキーワードで上位表示されている競合記事を確認:
- 競合がカバーしているが自社がカバーしていない悩みはあるか
- 競合のLLMO構造品質（まとめ/FAQ/構造化データ）はどうか
