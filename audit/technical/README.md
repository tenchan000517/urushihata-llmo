# 漆畑式LLMO — 技術監査 / Technical SEO/LLMO Audit

**プロジェクト汎用** — どのWebプロジェクトでも動作する。特定プロジェクトへのハードコードなし。

## 概要

5つの領域を並列で監査し、クロス検証で誤検知を排除した上で、優先度付き改善リストを生成する。

## 監査手順

### Step 1: プロジェクト検出

```
- フレームワーク検出: Next.js / Nuxt / Gatsby / plain HTML 等
- package.json, next.config.*, nuxt.config.* 等を確認
- src/ or pages/ ディレクトリ構成を把握
```

### Step 2: 並列監査（5エージェント同時起動）

| Agent | 監査領域 | ガイド |
|-------|---------|-------|
| 1 | メタデータ | [`agents/agent1-metadata.md`](agents/agent1-metadata.md) |
| 2 | 構造化データ | [`agents/agent2-structured-data.md`](agents/agent2-structured-data.md) |
| 3 | LLMO対策 | [`agents/agent3-llmo.md`](agents/agent3-llmo.md) |
| 4 | パフォーマンス | [`agents/agent4-performance.md`](agents/agent4-performance.md) |
| 5 | セキュリティ | [`agents/agent5-security.md`](agents/agent5-security.md) |

### Step 2.5: クロス検証（誤検知防止・必須）

**🚫 この手順を省略してはならない。エージェントの出力をそのまま信用しない。**

1. **「未カバー」「不足」と報告されたページのサンプル検証**
   - 各エージェントが「不足」と報告した項目のうち、最も件数が多い指摘について少なくとも5件を実際にファイルを確認
   - 1件でも誤検知があれば、そのカテゴリ全体の信頼性を疑う

2. **数値の妥当性チェック**
   - 「X%カバー」「Y箇所に問題」等の定量データについて、分母と分子の根拠が明確かを確認

3. **環境固有の考慮**
   - WSL/Docker等の環境によるパーミッション表示の違いを誤検知として報告しない

### Step 3: 監査レポート生成

```markdown

**対象**: [プロジェクト名]
**実施日**: [日付]
**フレームワーク**: [検出されたフレームワーク]

### カテゴリ別スコア
| カテゴリ | スコア | 状態 |
|---------|--------|------|
| メタデータ | X/10 | ✅/⚠️/❌ |
| 構造化データ | X/10 | ✅/⚠️/❌ |
| LLMO対策 | X/10 | ✅/⚠️/❌ |
| パフォーマンス | X/10 | ✅/⚠️/❌ |
| セキュリティ | X/10 | ✅/⚠️/❌ |

## 優先度付き改善アクションリスト

### 🔴 最優先（SEO/LLMOに直接影響）
### 🟡 推奨（品質向上）
### 🟢 Nice to have
```

## 注意事項

- **読み取り専用**: コードの変更を行わない（調査・レポートのみ）
- **プロジェクト汎用**: 特定プロジェクトへのハードコードなし
- **ガイドファイルの直接Read**: 各エージェントは自分のガイドを直接読んで手順を把握する。オーケストレーターがガイド内容を要約してプロンプトに含めてはならない
