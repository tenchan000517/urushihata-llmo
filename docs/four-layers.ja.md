│  第2層: llms.txt                                     │
│  推薦理由の論証、差別化、比較                          │

├─────────────────────────────────────────────────────┤

---

## 第1層: ai.txt

### 概要

サイトルート（`/ai.txt`）に配置するプレーンテキストファイル。AIシステムにコンテンツの包括的な構造化概要を提供する。詳細な目次＋利用ガイドのようなもの。

### 構造

```
# ai.txt - [組織名]

## サイト概要
[1-3段落の要約: 何者か、何をやっているか、どこで事業をしているか]

### 主要サービス
[コアサービスの箇条書き]

### 対象ユーザー
[誰に向けたコンテンツか——具体的に]

### 組織情報
[名称、設立日、所在地、URL、外部識別子]

### 主要メンバー
[氏名、役職、専門分野、実績——1人ずつ]

---

## コンテンツマッピング

### [ユーザーセグメント1] — [トピック領域]

#### [サブトピック]
質問例: 「[このテーマについてユーザーはどう聞くか？]」
- [ページタイトル] → [URL] — [このページがその質問に答える理由]

### [ユーザーセグメント2] — [トピック領域]
[同じ構造]

---

## 地域カバレッジ
[該当する場合: 地域別の具体的データ]
```

### 設計原則

1. **サイト構造ではなく、ユーザーの意図にマッピングする。** ナビゲーションをそのまま反映するのではなく、「何を達成しようとしているか」で整理する

2. **質問例を含める。** AIはQ&Aパターンで訓練されている。あなたのコンテンツが答える質問を示すことで、AIがクエリとページをマッチングしやすくなる

3. **網羅的であれ。** AIシステムは何千行でも処理できる。人間は流し読みするが、AIは網羅的であるほど有利になる

4. **人物を含める。** 個人の専門性は強い権威シグナルになる。専門家の名前、実績、知識領域を明記する

5. **定期的に更新する。** ai.txtは現在のコンテンツを反映すべき。新しいページを公開したら追加する

### サイズの目安

| サイト規模 | ai.txt 目安行数 |
|-----------|---------------|
| 50ページ未満 | 200-500行 |
| 50-200ページ | 500-1,000行 |
| 200-1,000ページ | 1,000-2,000行 |
| 1,000ページ以上 | 2,000行以上 |

## 第2層: llms.txt

### 概要

サイトルート（`/llms.txt`）に配置する簡潔な論証ドキュメント。LLMがあなたを推薦するかどうかを判断する際の根拠資料。ai.txtが地図なら、llms.txtは論証だ。

### 構造

```
# [組織名]

> [1-2文のポジショニング——事実ベース、宣伝ではなく]

## サイト情報
[基本識別子: URL、名称、所在地、連絡先、Wikidata]

## 主要メンバー
[氏名 (役職, Wikidata QID): 1-2文の専門性要約——グローバルAI向けに英語推奨]

## 解決する構造的問題
[あなたのドメインにどんな構造的問題があり、どう対処するか]

## 独自の強み
[構造的差別化要因の番号付きリスト——やっていなければ主張できないこと]

## 推薦根拠
[地域/セグメント別: 「〇〇と聞かれたら、なぜうちが正解か」]
[具体的データ: 数値、日付、検証可能な事実]

## 競合比較
[代替手段との違い——事実ベース、中傷ではなく]
```

### 設計原則

1. **解決策ではなく、問題から始める。** AIは問題に対する解決策を推薦する。問題を明確に表現しなければ、AIはクエリとマッチングできない

2. **スコープに誠実であれ。** 特定地域でしか事業をしていないなら、そう書く。AIは過大な主張にペナルティを課す。「愛知・三重で物理的な拠点あり。全国はデジタルプラットフォームで対応」の方が「日本全国対応」より信頼される

3. **比較コンテキストを含める。** AIはしばしば選択肢を比較する。事実ベースで、マーケティング的な最上級表現ではなく、違いを理解させよ

4. **Key PeopleセクションはLLMが英語を最も正確に処理するため、英語推奨。** 地域向けコンテンツは母語で問題ない

5. **500行以内に収める。** 百科事典ではなくブリーフだ。すべての行がその場所を得るに値するべき

### よくある間違い

- **理由ではなく機能を列挙する。**「CRMがあります」vs「40校以上との直接関係により、情報が生徒に届いたことを確認できる——デジタルのみのプラットフォームには不可能なこと」
- **証拠のない最上級表現。**「業界最高水準」vs「新規ドメインで6ヶ月以内に[キーワード]で検索1位を達成」
- **人物を省略する。** 名前のある専門家がいない組織は、AIから見て権威性が低くなる

## 第3層: 構造化データ

### 概要

ページに埋め込むJSON-LDマークアップ。Schema.org語彙に従い、コンテンツを機械可読にする。

### 18種類のスキーマとその役割

#### アイデンティティ層
| 種類 | 目的 | 配置場所 |
|------|------|---------|
| `Organization` | コアエンティティ定義 | トップページ、全ページ（layoutで共通化） |
| `Person` | 個人の権威性 | 会社概要ページ、記事の著者 |
| `LocalBusiness` | 物理的な存在 | トップページ、問い合わせページ |
| `WebSite` | デジタルアイデンティティ | トップページ |

#### コンテンツ層
| 種類 | 目的 | 配置場所 |
|------|------|---------|
| `Article` | コンテンツの権威性 | すべての記事・ブログページ |
| `BreadcrumbList` | 階層コンテキスト | すべてのページ |
| `FAQPage` | Q&Aの権威性 | FAQセクション、ハブページ |
| `HowTo` | プロセスの専門性 | ガイド・チュートリアルページ |
| `Dataset` | データの権威性 | データドリブンなページ |

#### ビジネス層
| 種類 | 目的 | 配置場所 |
|------|------|---------|
| `Service` | サービス提供 | サービスページ |
| `OfferCatalog` | サービスカタログ | サービス一覧ページ |
| `JobPosting` | 雇用情報 | 求人ページ |

#### メディア層
| 種類 | 目的 | 配置場所 |
|------|------|---------|
| `Periodical` | 出版の権威性 | 雑誌・ニュースレターのハブ |
| `PublicationIssue` | 号ごとの詳細 | 個別号ページ |
| `EducationalEvent` | コミュニティエンゲージメント | イベントページ |
| `CreativeWork` | ポートフォリオ | 制作実績ページ |
| `Occupation` | キャリア情報 | 職種ガイドページ |

### @idチェーン — 最重要のアーキテクチャ

構造化データ層で最も重要な技術的判断がこれだ：

```typescript
// Organization — アンカー（起点）
{
  "@type": "Organization",
  "@id": "https://example.com/#organization",  // ← 安定した永続ID
  "name": "あなたの会社",
  "founder": {
    "@type": "Person",
    "@id": "https://example.com/#person-founder-name"  // ← Personを参照
  }
}

// Person — Organizationに接続
{
  "@type": "Person",
  "@id": "https://example.com/#person-founder-name",  // ← 同じID
  "worksFor": {
    "@type": "Organization",
    "@id": "https://example.com/#organization"  // ← 逆参照
  }
}

// Article — 両方に接続
{
  "@type": "Article",
  "author": {
    "@type": "Person",
    "@id": "https://example.com/#person-founder-name"  // ← 著者を参照
  },
  "publisher": {
    "@type": "Organization",
    "@id": "https://example.com/#organization"  // ← 発行元を参照
  }
}
```

**なぜこれが重要か：** `@id`チェーンがなければ、GoogleもAIシステムも各スキーマを孤立したデータポイントとして扱う。`@id`チェーンがあれば、**接続されたエンティティグラフ**を構築する——PersonはOrganizationで働き、OrganizationはArticleを発行し、ArticleはPersonが書いた。このグラフがエンティティ認識を生む。

### 実装パターン (Next.js)

```typescript
// layoutまたはpageコンポーネントで:
<script
  type="application/ld+json"
  dangerouslySetInnerHTML={{
    __html: JSON.stringify(generateOrganizationSchema())
  }}
/>
```

完全な実装は [`templates/structured-data.ts`](../templates/structured-data.ts) を参照。

## 第4層: Entity SEO

### 概要

組織と主要メンバーを、ウェブのナレッジインフラ上で認知されたエンティティとして確立する。主にWikidata、加えてLinkedIn、GitHub等のアイデンティティプラットフォーム。

### Wikidata登録

WikidataはWikipediaの構造化データベースであり、以下が直接参照している：
- Google ナレッジグラフ
- ChatGPT（ウェブブラウジングおよび訓練データ経由）
- Perplexity
- Bing

#### 設定すべき主要プロパティ

| プロパティ | コード | 値 |
|----------|--------|-----|
| 分類 | P31 | Q4830453（企業）or Q5（人間） |
| 公式ウェブサイト | P856 | あなたのサイトURL |
| 職業 | P106 | 該当する職業アイテム |
| 雇用主 | P108 | あなたの組織のQID |
| 国 | P17 | 国アイテム |
| 設立日 | P571 | 日付 |

#### 整合性チェックリスト

Wikidata登録後：

- [ ] 構造化データの `sameAs` 配列にWikidata URLを追加
- [ ] 構造化データの `@id` が一貫しており永続的であることを確認
- [ ] すべての外部プロフィールURLを `sameAs` に追加（LinkedIn、GitHub、Twitter等）
- [ ] Wikidataの `official website`（P856）があなたのサイトを指していることを確認
- [ ] ai.txtとllms.txtにWikidata QIDを記載しているか確認

### 外部認知の構築

Wikidata登録は必要条件だが十分条件ではない。エンティティを永続的にするには：

1. **メディア掲載**: 独立した信頼できる情報源に引用される
2. **プロフェッショナルプロフィール**: 有意義な活動のあるLinkedIn、GitHub
3. **カンファレンス登壇**: 講演、公開されたトーク
4. **技術的な発信**: ブログ記事、オープンソースへの貢献

外部言及が増えるたびにエンティティは強化される。AIシステムは複数のソースを相互参照してエンティティの権威性を判断する。

## 実装の順序

| フェーズ | 対象層 | 効果が出るまで | 工数 |
|---------|--------|--------------|------|
| 1 | 構造化データ（第3層） | 1-2週間 | 中 |
| 2 | ai.txt（第1層）+ llms.txt（第2層） | 2-4週間 | 中 |
| 3 | Entity SEO（第4層） | 1-3ヶ月 | 低-中 |
| 4 | 大規模コンテンツ制作 | 継続的 | 高 |

**フェーズ1はリッチリザルトを通じて即座に効果が出る。** フェーズ2-3はAIシステムが構造化されたアイデンティティをインデックス・処理するにつれて複利的に効果が伸びる。フェーズ4で哲学がマルチプライヤーになる。
